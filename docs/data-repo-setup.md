# Setting up a JADE LENS data repo (v0.1.0)

In v0.1.0 the data-repo bootstrap is manual. Future versions will automate
this via a `jadelens init` flow; until then, this doc is the source of
truth for what a fresh data repo needs to contain.

The files listed below should be committed to the data repo. After that's
done, `cd` into the data repo and run `claude` — the session-start hook
takes care of installing `jadelens-apply` from the latest `main`, rendering
the skill, and optionally symlinking it into `~/.claude/skills/` for
global `/<name>` invocation.

## Layout overview

```
<data-repo>/
├── .claude/
│   ├── settings.json          # registers the SessionStart hook
│   └── hooks/
│       └── session-start      # executable; bootstraps every session
├── .jade/
│   └── config.json            # user + assistant names, shared across devices
├── .gitignore                 # ignores .claude/skills/ (the rendered skill)
└── index.json                 # bot-maintained map of primary files (starts empty)
```

`.jade/` and `.claude/` are off-limits to the bot (enforced by
`jadelens-apply`). `index.json` lives at the root because it *is*
bot-managed — see DESIGN.md §4.6.

## File contents

### `.jade/config.json`

Shared across all devices that read/write this data repo. The session-start
hook renders the skill from these values; updating them then deleting the
rendered skill (or running `jaderebuild`) picks up the new values.

```json
{
  "user": {
    "full_name": "<your full name>",
    "short_name": "<your short name / nickname>"
  },
  "assistant": {
    "name": "<assistant name — also the slash command, e.g. 'jade'>"
  }
}
```

All three fields are required. `jadelens render-skill` fails loud on
missing or empty fields rather than rendering `{{undefined}}` placeholders.

### `index.json` (at the data repo root)

Starts empty; the bot adds entries as it creates primary files.

```json
{}
```

### `.gitignore`

At minimum:

```gitignore
# Rendered skill files — regenerated by the session-start hook.
.claude/skills/
```

Ignoring the whole `.claude/skills/` dir (rather than just one assistant's
subdirectory) means renaming the assistant doesn't leave a stale tracked
subdir behind. Add your own personal ignores below as needed.

### `.claude/settings.json`

Registers the `SessionStart` hook. `$CLAUDE_PROJECT_DIR` is set by Claude
Code to the project root, so the path is robust to invocation cwd.

```json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/session-start"
          }
        ]
      }
    ]
  }
}
```

### `.claude/hooks/session-start`

Executable. Set the bit *before* `git add`, or fix after with
`git update-index --chmod=+x .claude/hooks/session-start`.

```bash
#!/usr/bin/env bash
# Renders the JADE LENS skill from the bundled template every session,
# installing `jadelens-apply` first if it isn't already on PATH.
#
# Idempotent: if jadelens-apply is already present (e.g. the developer's
# `uv tool install -e` editable install on desktop), the install step
# no-ops. If the rendered SKILL.md already exists, render-skill no-ops
# too. Delete the rendered skill to force a re-render.

DATA_REPO="$(cd "$(dirname "${BASH_SOURCE[0]}")/../.." && pwd)"

# 1. Ensure jadelens-apply is on PATH. Pinned to main during early
#    development; once data versions track code versions, this will read
#    from .jade/version instead.
if ! command -v jadelens-apply >/dev/null 2>&1; then
    uv tool install "git+https://github.com/tomasssalles/jade-lens.git@main" || {
        echo "session-start: failed to install jadelens; cannot render skill" >&2
        exit 1
    }
fi

# 2. Render the skill into <data-repo>/.claude/skills/<assistant.name>/SKILL.md.
jadelens render-skill "$DATA_REPO" || {
    echo "session-start: jadelens render-skill failed" >&2
    exit 1
}

# 3. Symlink ~/.claude/skills/<assistant.name>/ → the rendered skill dir,
#    so /<name> works from any cwd. On claude.ai the symlink doesn't
#    persist; that's fine — the same dir is reachable via the project's
#    own .claude/skills/<name>/ at session start.
ASSISTANT_NAME=$(
    python3 -c "import json; print(json.load(open('$DATA_REPO/.jade/config.json'))['assistant']['name'])" \
        2>/dev/null
) || exit 0

HOME_SKILLS="$HOME/.claude/skills"
HOME_SKILL="$HOME_SKILLS/$ASSISTANT_NAME"
DATA_SKILL="$DATA_REPO/.claude/skills/$ASSISTANT_NAME"

# Already pointing at the right place? Silent no-op (the steady state).
if [ -L "$HOME_SKILL" ] && [ "$(readlink "$HOME_SKILL")" = "$DATA_SKILL" ]; then
    exit 0
fi

if [ -e "$HOME_SKILL" ] || [ -L "$HOME_SKILL" ]; then
    # Something else is there. Don't clobber; tell the user how to take over.
    echo "Note: $HOME_SKILL already exists but isn't pointing at this data repo."
    echo "If you'd like /$ASSISTANT_NAME (from any directory) to point here, inspect"
    echo "the existing path and then run:"
    echo "  rm -rf '$HOME_SKILL' && ln -s '$DATA_SKILL' '$HOME_SKILL'"
    exit 0
fi

# Path is empty. Try to create the parent dir + symlink in one shot.
if mkdir -p "$HOME_SKILLS" 2>/dev/null && ln -s "$DATA_SKILL" "$HOME_SKILL" 2>/dev/null; then
    echo "✓ Symlinked $HOME_SKILL → $DATA_SKILL"
    echo "  You can now use /$ASSISTANT_NAME from any Claude Code session, in any directory."
else
    echo "To enable /$ASSISTANT_NAME from any Claude Code session (not just inside this data repo), run:"
    echo "  mkdir -p '$HOME_SKILLS' && ln -s '$DATA_SKILL' '$HOME_SKILL'"
fi
```

The hardcoded GitHub URL (`tomasssalles/jade-lens`) points at the
canonical deployment. Self-hosters who maintain their own fork should
change it to their own `git+https://github.com/<owner>/<repo>.git@<ref>`.

## Setup steps

1. Create a private GitHub repo for your data; clone it locally.
2. Create the five files above with the contents shown. Fill in the
   placeholders in `.jade/config.json`.
3. Make the hook executable: `chmod +x .claude/hooks/session-start`.
4. Commit and push.
5. `cd` into the data repo and run `claude`. The hook will install
   `jadelens-apply` if missing, render the skill into
   `.claude/skills/<assistant.name>/SKILL.md`, and then either symlink
   it into `~/.claude/skills/` for global `/<name>` access (if the
   parent dir is writable and nothing's in the way), or print the exact
   `mkdir -p && ln -s` command for you to run yourself.
6. (Optional — desktop dev) `uv tool install -e /path/to/jade-lens` if
   you're working on the code repo and want the editable install rather
   than the canonical `@main` one the hook would install.

## Future automation: defaults for `jadelens init`

When the bootstrap is automated (likely a `jadelens init <data-repo>`
subcommand parallel to the existing onboarding flow in `jadelens/cli.py`),
the user-prompt step should re-use the defaults the current onboarding
flow already settled on:

- **`user.full_name`** — default to the data repo's
  `git config user.name`, if set. Falls back to no default (prompt
  required) if absent.
- **`user.short_name`** — default to the first whitespace-separated
  token of the chosen full name (e.g. `"Tomás Silveira Salles"` →
  `"Tomás"`).
- **`assistant.name`** — default to `jade`.

These match the prompts in `jadelens.cli` (`prompt_user_full_name`,
`prompt_user_short_name`, `prompt_assistant_name`) — the same code can
be lifted and adapted into the init flow. For non-interactive callers
(scripts, claude.ai), the same derivation should apply silently when a
field is absent from input.

## Keeping this doc in sync

This is the source of truth for future automation (`jadelens init` and
similar). When the data-repo shape changes (new required file, schema
update, hook revision), update this doc *with the new contents*; the
