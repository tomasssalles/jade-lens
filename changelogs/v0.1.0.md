# What's new in v0.1.0?

_This is the very first version!_ 🎉

_It is meant to test how well the bot autonomously organizes the data in a structured manner given chaotic input. Most of the convenience code is left out and a lot of manual work is required to use it._

- **Claude Code skill, rendered into the data repo by a session-start hook** (DESIGN.md §12.7). The skill lives at `<data-repo>/.claude/skills/<assistant.name>/SKILL.md` and is rendered fresh on every session start by `<data-repo>/.claude/hooks/session-start` (both committed to the data repo). Claude Code auto-discovers it from the repo root on any surface — desktop TUI, claude.ai browser, claude.ai mobile. On desktop the hook also tries to symlink `~/.claude/skills/<assistant.name>/` to the rendered dir for global `/<name>` invocation from any cwd; if it can't, it prints the exact `mkdir -p && ln -s` command for the user to run. The SKILL.md body describes the data conventions and the `jadelens-apply` protocol; native Claude Code Read / Grep / Glob are used for discovery.
- **`jadelens-apply` as a `uv tool install`-able console script.** Exposed by the `jadelens` Python package; the session-start hook installs from GitHub if missing (pinned to `@main` during early development). The hook IS the install path — no separate `install` command to run by hand. Skill templates are bundled inside the package as resources, so the same `uv tool install` brings both the runtime and the latest template version with it.
- **Shared config in `<data-repo>/.jade/config.json`** (snake_case keys: `user.full_name`, `user.short_name`, `assistant.name`). Committed and shared across all the user's devices, so the rendered skill is consistent everywhere. `jadelens render-skill <data-repo>` reads this file, picks the highest-version bundled template, and writes the skill — failing loud on missing or empty fields rather than emitting `{{undefined}}` placeholders.
- **Index at `<data-repo>/index.json`** (data repo root, not under `.jade/`). The bot maintains it; it lives at the root because the bot writes it and the bot is forbidden from touching `.jade/`.
- **Protected paths.** The bot must not read from or write to any top-level path whose name starts with `.` (`.claude/`, `.git/`, `.gitignore`, `.jade/`, future dotfiles). Enforced in `jadelens-apply` at op-parse time AND prescribed by the SKILL.md body.
- **Five-operation mutation set** (DESIGN.md §4.2), passed to `jadelens-apply` as a typed list:
  - `json_patch` — RFC 6902 patch against a `.json` file.
  - `unified_diff` — unified diff against a non-`.json` file (markdown in practice); 0 context lines default.
  - `create_file` — new file with initial content. Path must end in `.json` or `.md` (the *editable file suffixes*; the set is extensible). Missing parent directories are auto-created (`mkdir -p`). `.json` content is parse-validated at parse time, so an invalid JSON file can't be created and only blow up later. Typo risk in paths is an accepted v0.1.0 risk.
  - `delete_path` — recursive delete of a file or directory. Wikilink references to the target are checked *after the batch has applied*; if any survive, the batch is rejected and reverted.
  - `rename_path` — file or directory rename. Source must exist; target must not. For file renames the suffix must be preserved (no `notes.md → notes.json`). Wikilink references elsewhere are auto-rewritten by the runtime as part of the post-apply pass.
- **Parse-time and apply-time validation**, fail-fast at the earliest possible stage:
  - Path-suffix rules: `json_patch` only on `.json`, `unified_diff` not on `.json`, `create_file` in the editable-suffix set, `rename_path` preserves file suffix.
  - Protected-path rule (above) rejects any op targeting a top-level dot-prefixed path.
  - Structural validation per op: required keys exact, no extras (catches realistic bot mistakes like forgetting to wrap a single JSON-Patch op in a list).
  - `create_file` with `.json` path: content must parse as valid JSON at parse time.
  - `unified_diff`: claimed line content must match the file at the claimed line number at apply time.
  - `json_patch`: standard RFC 6902 validation at apply time (test/remove/replace value checks; missing paths raise).
  - Any failure aborts the whole atomic batch — no partial application. The data repo reverts to HEAD via `git reset --hard && git clean -fd`.
- **Clean-tree precondition + atomic apply with revert on failure.** The workflow refuses to start if the data repo has uncommitted changes (which is what lets the revert be safe — we never clobber the user's in-flight manual edits). If anything fails mid-batch, the working tree is reset to HEAD and the batch is reported as a single failure to the bot. SKILL.md tells the bot **not to pre-check** with `git status` — the tool's own check is authoritative and reports cleanly on failure.
- **Atomic-data-change pipeline** (DESIGN.md §7.1): every `jadelens-apply` invocation that produces any operations → exactly **one git commit + one log entry**. Pure-query invocations produce nothing.
- **Operations log** at `<data-repo>/.jade/operations-log/<version>.jsonl` (per-version directory so migrations start a fresh file while leaving old entries readable). Append-only JSONL. Schema per line: `{"ts": "...", "commit_message": "...", "operations": [<op>, ...]}`. No prompt, no response text, no commit SHA — commit identity is recoverable from git (the log-line ↔ commit bijection). The commit_message IS inlined deliberately, keeping the log self-sufficient as the canonical audit record (forward-compat with a future move off git). Runtime is the only writer.
- **Bot writes the commit message** as part of the tool call — concise, one line, summarising intent. Verbatim user prompts are NOT repeated (output-token cost matters even on Pro, and the in-context prompt is rarely meaningful without the surrounding Claude Code chat). The commit_message lives both in git's commit message and in the corresponding ops-log entry.
- **Wikilink convention for path references** (DESIGN.md §4.3). Every reference from data to a data-repo file is written as `[[path]]` (paths are repo-root-relative) in both markdown files and JSON string values. Click-navigation in plain IDEs doesn't work; accepted tradeoff for rename safety. Wikilink consistency is enforced via a **deferred post-apply pass**: after every op in the batch has applied, the runtime rewrites references for each `rename_path` and verifies no dangling references remain for each `delete_path`. The bot can interleave clean-up ops in any order. Scope is git-visible files only — gitignored content is never touched. Three SKILL.md prescriptions for the bot: use `[[path]]` for any data-repo reference; do NOT manually rewrite refs on rename (runtime handles it); DO clear refs around delete (or unlink them as plain prose).
- **Index file maintained by the bot.** The bot updates `index.json` when it adds, removes, or restructures primary files, providing the human (and itself) with a navigational map. No runtime-enforced `alwaysLoad` mechanics in v0.1.0 — Claude Code does agentic discovery via its native Read / Grep / Glob, so the index is a navigational aid, not a discovery driver.
- **Forward-only correction** (DESIGN.md §7.4). When the user spots a mistake, they tell the bot in natural language. The bot reads context, fixes the data, and propagates the fix to any downstream references. No rewind, no replay.
- **Tool result echoes the applied operations** (DESIGN.md §12.4). After `jadelens-apply` applies a batch, the runtime returns a formatted reflection in the tool result so the user can see exactly what was changed inline in the TUI: commit-message header, then per-op blocks — `json_patch` pretty-printed (indent=2), `unified_diff` raw, `create_file` with byte count + raw content, `delete_path` one-line, `rename_path` with `→`. No ANSI styling in v0.1.0 — see DESIGN.md §12.4 for the future option, and for the future "side-effects" expansion (wikilink rewrites etc. that are currently invisible in the reflection).

## What's kept simple for now

- No web app (beyond a minimal landing page).
- Manual creation of the data repo, including the five v0.1.0 bootstrap files described in `docs/data-repo-setup.md` (`.jade/config.json`, `index.json`, `.gitignore`, `.claude/settings.json`, `.claude/hooks/session-start`). A future `jadelens init <data-repo>` will automate this.
- Manual syncing (`git pull/push` in the local data-repo clone).
- No data types with special handling or predefined shape/schema (e.g. calendar events, tasks, contacts).
- No code for controlled/targeted discoverability or prompt cache efficiency.
- No replay (and no rewind). Forward-only correction is the only correction mode.
- No automatic version comparison between data and code (the full §14 migration framework is deferred). The hook hardcodes `@main`; new code is picked up on the next cold claude.ai session, or via `git pull` on desktop with `uv tool install -e <local clone>`.
- No optional runtime safety net for the wikilink convention (e.g. flagging path-looking strings outside wikilinks). SKILL.md is prescriptive; non-compliance is fixed manually.
- No automatic inline-vs-sidecar promotion (DESIGN.md §4.4). The bot makes all promotion decisions explicitly via the five-op set — `create_file` for the new `.md`, `json_patch` to write the wikilink, optional index update.
- No user-configurable bot behaviour (terseness, response style). The skill template bakes in the project's opinions; user-driven customisation via `.jade/config.json` is future work (DESIGN.md §15.2).
- Symlink to `~/.claude/skills/<name>/` is best-effort. If the hook can't create it (parent missing, not writable, slot occupied), it prints the manual `mkdir -p && ln -s` command for the user; the skill still works from inside the data repo either way.
