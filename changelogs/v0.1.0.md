# What's new in v0.1.0?

This is the very first version! 🎉

The plan is to test how well the bot autonomously organizes the data in a structured manner given chaotic input. Most of the convenience code will be left out and a lot of manual work will be required.

## What's in scope

- **Claude Code `/jade` skill** installed at `~/.claude/skills/<name>/SKILL.md` (auto-discovered by every session — no aliases, no `--add-dir`). The installation machinery is in place; the SKILL.md body that describes the data conventions and the `handle_bot_response` protocol to the bot is the next milestone (currently a hello-world placeholder). Native Claude Code Read / Grep / Glob for discovery are fine and expected.
- **Installer and update commands** (DESIGN.md §12.7). The skill is rendered from a versioned template (`templates/skill/v<version>.md`) into the user's per-user skills dir; paths are baked in as concrete inline strings so the bot needs zero indirection at runtime. A `jade-lens-skill template-version=<v>` marker comment lets the update command find and rebuild installs by regex-extracting current config from the rendered file. Supports multi-install (different data repos) and skill rename (old install deleted after re-render). User-chosen skill name supported from day one.
- **Five-operation mutation set** (DESIGN.md §4.2), passed to `handle_bot_response` as a typed list:
  - `json_patch` — RFC 6902 patch against a `.json` file.
  - `unified_diff` — unified diff against a non-`.json` file (markdown in practice); 0 context lines default.
  - `create_file` — new file with initial content. Path must end in `.json` or `.md` (the *editable file suffixes*; the set is extensible). Missing parent directories are auto-created (`mkdir -p`). `.json` content is parse-validated at parse time, so an invalid JSON file can't be created and only blow up later. Typo risk in paths is an accepted v0.1.0 risk.
  - `delete_path` — recursive delete of a file or directory. Wikilink references to the target are checked *after the batch has applied*; if any survive, the batch is rejected and reverted.
  - `rename_path` — file or directory rename. Source must exist; target must not. For file renames the suffix must be preserved (no `notes.md → notes.json`). Wikilink references elsewhere are auto-rewritten by the runtime as part of the post-apply pass.
- **Parse-time and apply-time validation**, fail-fast at the earliest possible stage:
  - Path-suffix rules: `json_patch` only on `.json`, `unified_diff` not on `.json`, `create_file` in the editable-suffix set, `rename_path` preserves file suffix.
  - Structural validation per op: required keys exact, no extras (catches realistic bot mistakes like forgetting to wrap a single JSON-Patch op in a list).
  - `create_file` with `.json` path: content must parse as valid JSON at parse time.
  - `unified_diff`: claimed line content must match the file at the claimed line number at apply time.
  - `json_patch`: standard RFC 6902 validation at apply time (test/remove/replace value checks; missing paths raise).
  - Any failure aborts the whole atomic batch — no partial application. The data repo reverts to HEAD via `git reset --hard && git clean -fd`.
- **Clean-tree precondition + atomic apply with revert on failure.** The workflow refuses to start if the data repo has uncommitted changes (which is what lets the revert be safe — we never clobber the user's in-flight manual edits). If anything fails mid-batch, the working tree is reset to HEAD and the batch is reported as a single failure to the bot.
- **Atomic-data-change pipeline** (DESIGN.md §7.1): every `handle_bot_response` invocation that produces any operations → exactly **one git commit + one log entry**. Pure-query invocations produce nothing.
- **Operations log** at `.jade/operations-log/<version>.jsonl` in the data repo (per-version directory so migrations start a fresh file while leaving old entries readable). Append-only JSONL. Schema per line: `{"ts": "...", "commit_message": "...", "operations": [<op>, ...]}`. No prompt, no response text, no commit SHA — commit identity is recoverable from git (the log-line ↔ commit bijection). The commit_message IS inlined deliberately, keeping the log self-sufficient as the canonical audit record (forward-compat with a future move off git). Runtime is the only writer.
- **Bot writes the commit message** as part of the tool call — concise, one line, summarising intent. Verbatim user prompts are NOT repeated (output-token cost matters even on Pro, and the in-context prompt is rarely meaningful without the surrounding Claude Code chat). The commit_message lives both in git's commit message and in the corresponding ops-log entry. The web app will use a different commit-message strategy when it lands.
- **Wikilink convention for path references** (DESIGN.md §4.3). Every reference from data to a data-repo file is written as `[[path]]` (paths are repo-root-relative) in both markdown files and JSON string values. Click-navigation in plain IDEs doesn't work; accepted tradeoff for rename safety. Wikilink consistency is enforced via a **deferred post-apply pass**: after every op in the batch has applied, the runtime rewrites references for each `rename_path` and verifies no dangling references remain for each `delete_path`. The bot can interleave clean-up ops in any order (clean-up via `unified_diff` later in the batch satisfies the check; the scan only cares about end state). Scope is git-visible files only — gitignored content is never touched. Three SKILL.md prescriptions for the bot: use `[[path]]` for any data-repo reference; do NOT manually rewrite refs on rename (runtime handles it, saves output tokens); DO clear refs around delete (or unlink them as plain prose).
- **Index file maintained by the bot.** The bot updates the index when it adds, removes, or restructures primary files, providing the human (and itself) with a navigational map. No runtime-enforced `alwaysLoad` mechanics in v0.1.0 — Claude Code does agentic discovery via its native Read / Grep / Glob, so the index is a navigational aid, not a discovery driver.
- **Forward-only correction** (DESIGN.md §7.4). When the user spots a mistake, they tell the bot in natural language. The bot reads context, fixes the data, and propagates the fix to any downstream references. No rewind, no replay.
- **Tool result echoes the applied operations** (DESIGN.md §12.4). After `handle_bot_response` applies a batch, the runtime returns a formatted reflection in the tool result so the user can see exactly what was changed inline in the TUI: commit-message header, then per-op blocks — `json_patch` pretty-printed (indent=2), `unified_diff` raw, `create_file` with byte count + raw content, `delete_path` one-line, `rename_path` with `→`. No ANSI styling in v0.1.0 — see DESIGN.md §12.4 for the future option, and for the future "side-effects" expansion (wikilink rewrites etc. that are currently invisible in the reflection).
- **`jadelens` CLI startup nudge.** Before any flow, the CLI does a best-effort `git fetch` + commits-behind check against the code repo's origin (short timeout, silent on network failure). If the user's clone is behind, prints a one-line "N new commits on origin; cd to the code repo and `git pull && jadelens` to apply" and offers to abort. Never modifies code itself; the user controls when to update.
- **Calendar events handled via manual chat-paste** into freeform bot-designed records — calendar is not skipped, just deferred from its first-class §10 form to its v1-phase form. No API integration.

## What's kept simple for now

- No web app.
- Manual creation of the data repo, manual local cloning.
- No on-disk config file — config values live only in the rendered installed skill (DESIGN.md §12.7).
- Manually added version file and index in the data repo.
- Manual syncing (`git pull/push` in the local data-repo clone).
- No data types with special handling or predefined shape/schema (e.g. calendar events, tasks, contacts).
- No code for controlled/targeted discoverability or prompt cache efficiency.
- No replay (and no rewind). Forward-only correction is the only correction mode.
- No automatic version comparison between data and code (the full §14 migration framework is deferred); code updates require the user to run the `update` command from the code repo.
- No optional runtime safety net for the wikilink convention (e.g. flagging path-looking strings outside wikilinks). SKILL.md is prescriptive; non-compliance is fixed manually.
- **No automatic inline-vs-sidecar promotion** (DESIGN.md §4.4). The bot makes all promotion decisions explicitly via the five-op set — `create_file` for the new `.md`, `json_patch` to write the wikilink, optional index update. Automatic promotion (with its indexing-default nuance) lands in a later iteration.
